# ADR-0015 — Quality harness: test coverage floor

- Status: accepted
- Date: 2026-04-21

## Context

Vitest already produces v8-provider coverage on `npm run test:coverage`,
but the data is informational — no thresholds, no gating. The clean-code
arm (ADR-0007/0008/0009/0010) catches dead code, duplication, naming
drift, and code smells, but none of those sensors notice when a refactor
deletes the only test exercising a critical accessor in `src/lib/data.ts`.

The harness principle (ADR-0006) is *every invariant gets a sensor at
the cheapest enforcement point*. Test coverage is an invariant on the
**verification surface**, not the code. It belongs in `check:quality`
because it gates "the test suite still meaningfully exercises the code
graph", not "the code graph itself is well-shaped".

The trap with coverage gates is aspirational thresholds. Setting
`lines: 80` against a current 56% baseline turns the sensor into a
forever-red ratchet that nobody reads. ADR-0008 documented the same
trap for jscpd and resolved it by setting the threshold *at current
reality* (3% to fit the 0.89% baseline plus headroom). The same shape
applies here.

## Decision

Wire `check:quality:coverage` as `vitest run --coverage` and configure
deterministic floors in `packages/techradar/vitest.config.ts`:

```ts
coverage: {
  provider: "v8",
  include: ["src/**/*.{ts,tsx}"],
  exclude: [
    "src/**/*.test.*", "src/test/**", "packages/techradar/src/components/Icons/**",
    "packages/techradar/src/pages/_document.tsx", "packages/techradar/src/pages/_app.tsx",
  ],
  thresholds: {
    lines: 55, statements: 55, branches: 55, functions: 60,
  },
}
```

The thresholds match the current measured baseline:

- Statements 55.79% → floor 55
- Branches 58.37% → floor 55
- Functions 65.66% → floor 60
- Lines 56.78% → floor 55

These are **floors, not targets**. Bumping a floor is a deliberate
diffable act: a contributor who adds a feature with strong test
coverage can raise the corresponding floor in the same PR, locking in
the gain. A refactor that drops a metric below its floor fails the
gate, forcing an explicit decision.

Excludes are the existing reality:

- `*.test.*` and `src/test/**` — test infrastructure, not under test
- `packages/techradar/src/components/Icons/**` — generated by `@svgr/cli` (see
  `packages/techradar/AGENTS.md` 'Pitfalls')
- `packages/techradar/src/pages/_document.tsx` and `packages/techradar/src/pages/_app.tsx` — Next.js framework files
  with no business logic worth covering

### Rejected alternatives

- **Aspirational thresholds (e.g. lines 80%).** Rejected: forever-red
  gates are ignored. ADR-0008 documents this anti-pattern.
- **Per-file thresholds** (`thresholds: { 'src/lib/data.ts': { lines: 90 } }`).
  Rejected: brittle to refactors and renames; turns the config into a
  manifest. The aggregate floor catches mass deletions; per-file
  policy is review-time.
- **Coverage as informational-only (no gate).** Rejected: identical to
  the status quo. The whole point of `check:quality` is that every
  sensor in the umbrella gates. An informational coverage report is
  what `npm run test:coverage` already provides.
- **Use Istanbul provider instead of v8.** Rejected: v8 is faster,
  built-in to the runtime, and matches what `npm run test:coverage`
  already produces. Istanbul's instrumentation overhead buys nothing
  here.

## Consequences

- `check:quality` grows from four to five sub-scripts. DoD command set
  is unchanged in shape.
- Coverage runs on every full DoD pass. Wall-clock cost is a single
  vitest pass with `--coverage` (~5–10s on this codebase).
- The floors are deliberately lax. The first time a contributor adds
  a high-coverage feature, they should raise the matching floor in
  the same PR. This is an explicit handshake, not an automated
  ratchet.
- `check:quality:coverage` and `test:coverage` overlap. The latter
  stays as the dev-loop alias (no failure on threshold miss when run
  manually); the former is the gated form. Vitest applies thresholds
  only when coverage is enabled, so `npm test` (without
  `--coverage`) is unaffected.
