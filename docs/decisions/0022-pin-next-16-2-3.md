# ADR-0022 — Pin `next` to `16.2.3` (no caret) until upstream regression #92580 is fixed

- Status: superseded by ADR-0023
- Date: 2026-04-22

## Context

Consumers of `@porscheofficial/porschedigital-technology-radar@1.2.1` reported
that `npx techradar build` fails during the static-export prerender phase with:

```
TypeError: Cannot read properties of null (reading 'useContext')
    at ignore-listed frames
Error occurred prerendering page "/[quadrant]/[id]: /<…>/<…>"
```

The error fires on **every** dynamic detail page (`/[quadrant]/[id]`),
across all quadrants, with no application-level frames in the stack. Index,
history, 404, and quadrant pages prerender fine.

### Reproduction & isolation

Tested in `/Users/patrice.bouillet/git/cs-technology-radar` (a real consumer
with 172 radar items → 183 prerendered pages, vanilla `npm`, no `pnpm`):

- `next@16.2.3` → static export succeeds.
- `next@16.2.4` → every `/[quadrant]/[id]` page fails identically.

Our own development radar (27 items → 38 pages) does **not** trip the bug,
which is why CI and `pnpm run build` in this repo stay green. The regression
appears to need fan-out across enough parallel prerender workers (Next 16
defaults to 11) to manifest.

### Root cause (upstream)

Diffing the `next-16.2.3.tgz` and `next-16.2.4.tgz` tarballs from npm shows
that the only Pages-Router-touching change in the 16.2.3 → 16.2.4 range is:

> **Scope Safari `?ts=` cache-buster to CSS/font assets only (Pages Router)**
> ([vercel/next.js#92580](https://github.com/vercel/next.js/pull/92580),
> committed 2026-04-14)

That PR rewrote `renderToHTMLImpl` in `dist/server/render.js` (1778-line
diff), reshaped `HtmlProps` (the value carried by `HtmlContext`), and
republished both `dist/compiled/next-server/pages.runtime.prod.js` and
`dist/compiled/next-server/pages-turbo.runtime.prod.js`. The remaining 8
commits in 16.2.4 are unrelated (Turbopack `reqwest` bump, fs-watcher fix,
compiler boolean defines, etc.) and do not touch the Pages Router or the
React runtime.

The `useContext` null is therefore almost certainly a regression in #92580
where some code path in the Pages runtime now reaches `React.useContext`
before the React reference is initialised in the prerender worker.

An upstream issue should be filed citing #92580 directly.

## Decision

Pin `next` to **exactly `16.2.3`** in `package.json#dependencies` (no caret)
until upstream ships a fix.

```diff
-    "next": "^16.2.3",
+    "next": "16.2.3",
```

This is intentional, narrowly-scoped, and reversible.

### Why pin exactly, not range-cap

- `<16.2.4` would still let `^16.2.3` resolve to any future `16.2.5+` patch
  that may carry the same regression unchanged.
- An exact pin produces a single, diffable bump when we relax it. The act of
  removing the pin is the same act as confirming the upstream fix.

### Why we still ship the package now

- The published tarball has `next` in `dependencies` (per ADR-0021), so
  consumer installs honor the exact pin.
- All other CLI machinery (shadow workspace, sanitization, recategorized
  deps) is unchanged.
- 1.2.2 is a one-line `package.json` change + this ADR. No code, no test, no
  build-pipeline change.

## Alternatives considered

**A. Wait for upstream fix, no pin.** Rejected — every `npx techradar build`
in a real-size consumer keeps failing in the meantime. Downstream pain is
the whole point of fixing this.

**B. Cap to `<16.2.4`** (e.g. `>=16.2.3 <16.3.0`). Rejected — accepts the
risk that `16.2.5` (or `16.2.4-canary.*`) silently carries the regression.
An exact pin is louder.

**C. Patch around it in our code** (workaround in `_document` or page
components). Rejected — the failure is inside Next's compiled runtime
("ignore-listed frames"). There is no application-level workaround.

**D. Switch to App Router.** Rejected — out of scope, multi-week change, and
ADR-0001 documents why we chose Pages Router.

## What does NOT change

- `react`, `react-dom` stay on `^19.2.5` (caret) — the bug is in Next, not
  React.
- The shadow-workspace install pipeline (ADR-0021) is unchanged. Consumers
  still get a clean `npm install` with no peer-dep conflicts.
- Maintainer Definition of Done is unchanged. `pnpm run build` still works
  on `16.2.3` and on any future patched version (once we relax the pin).
- No code changes. No test changes. No harness changes.

## Consequences

**Positive**

- Consumers installing `@porscheofficial/porschedigital-technology-radar@1.2.2`
  get a Next that actually prerenders.
- The pin is a single line; removing it is a one-line PR + ADR-0023
  superseding this one.

**Negative**

- We forgo any unrelated bug fixes shipped in 16.2.4+ until we relax the
  pin. Acceptable given the alternative is "build fails for everyone".
- New consumers who already have `next@16.2.4+` in their root project will
  see npm resolve our `dependencies` to `16.2.3` and theirs to `16.2.4+`.
  This is fine because the shadow workspace at `.techradar/` installs from
  our pinned `package.json`, not the consumer's root `node_modules`.

**Followups (not part of this ADR)**

- File the upstream issue at vercel/next.js with a link to #92580 and the
  cs-technology-radar reproduction.
- Once upstream patches, supersede this ADR with one that relaxes the pin
  and records the fixed version.
