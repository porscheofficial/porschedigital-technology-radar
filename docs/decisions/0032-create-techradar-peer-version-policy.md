# ADR-0032 — `create-techradar` peer-version policy

- Status: accepted
- Date: 2026-05-07

## Context

ADR-0030 established `create-techradar` as a one-shot bootstrapper rather than a template owner. That ADR explicitly decided to query the npm registry for the `latest` tag of `@porscheofficial/porschedigital-technology-radar` and write a caret range (`^X.Y.Z`) into the generated `package.json`. The rationale was to avoid a release-please coupling between the two packages and to prevent silently locking new users to old framework versions when the scaffolder is not republished.

However, writing a caret range for a framework dependency can lead to unpredictable behavior. When a user runs `npm install` months later, they may pull in a minor or patch update that introduces subtle regressions or requires a template migration that the scaffolder did not provide. The starter files generated by `techradar init` at scaffold time are intimately coupled with the framework version active at that moment.

To ensure a highly predictable, reproducible baseline for consumers, we need a stricter versioning policy where a scaffolded project is pinned to the exact framework version.

## Decision

`create-techradar` scaffolds a project pinned to the EXACT framework version at scaffold time, omitting the caret or tilde prefix (e.g., `"@porscheofficial/porschedigital-technology-radar": "2.1.0"`, not `"^2.1.0"`). 

Users must opt-in to framework upgrades explicitly (e.g., via `npm install @porscheofficial/porschedigital-technology-radar@latest`). 

As a consequence of this exact pinning, `create-techradar` releases must be coordinated with framework releases. We will configure the release pipeline to publish the two packages in lockstep, guaranteeing that each `create-techradar` version maps predictably to its peer framework version.

## Consequences

### Good

- Consumers get a fully reproducible environment. Running `npm install` on a fresh clone of a scaffolded project months later will install the exact framework version that originally generated the template files.
- The risk of "silent drift" where a minor update breaks user-modified template files is eliminated.
- Upgrades become deliberate, conscious actions by the consumer, accompanied by reading the changelog.

### Trade-offs / risks

- Reverses the ADR-0030 decision regarding caret ranges and decoupling.
- Requires coordinating releases for the two packages. A release-please coupling must be maintained to ensure `create-techradar` stays synchronized with `@porscheofficial/porschedigital-technology-radar` versions.
- If we forget to publish the scaffolder alongside the framework, new users scaffolding with the latest `create-techradar` might get a slightly older pinned version. Lockstep releases mitigate this.
